# 構想とアーキテクチャ

## 1. 課題と目的

社内環境でSASを利用しているチームの多くは、その出力先としてExcel帳票を採用している。
Excel帳票は導入が手軽な反面、以下の課題に直面する。

- 同時参照性が低い（ファイル排他、メール添付による配布の煩雑さ）
- 最新性の担保が手間（更新ごとに再配布が必要）
- 可視化表現が貧弱、対話操作ができない
- 帳票が増えていくとフォルダ管理が破綻する

一方で、本格的なBIツール（Tableau / Power BI / SAS Visual Analytics 等）の導入はコスト・承認プロセス・運用負荷の壁が大きい。

本リポジトリは、その間を埋める「**追加ソフトなしで動くダッシュボード**」を、PowerShellとHTML/JSのみで構成するレシピを提供する。

## 2. 設計思想

### 2.1 関心の分離（DDDの考え方を借りる）

```
┌────────────────────────────────────────────────┐
│  Interface 層（ブラウザUI）                      │
│  - HTML/CSS/JS                                  │
│  - 可視化（Chart.js等）                         │
│  - フィルタ・絞り込みUI                          │
└─────────────────┬──────────────────────────────┘
                  │  HTTP / 静的ファイル
┌─────────────────▼──────────────────────────────┐
│  Application 層（ユースケース）                  │
│  - ダッシュボード取得                            │
│  - 指標の絞り込み                                │
└─────────────────┬──────────────────────────────┘
                  │
┌─────────────────▼──────────────────────────────┐
│  Domain 層（コア）                              │
│  - 帳票 / ダッシュボード / 指標 / 断面          │
│  - 集計ルール（SAS非依存に再表現）              │
└─────────────────┬──────────────────────────────┘
                  │
┌─────────────────▼──────────────────────────────┐
│  Infrastructure 層                              │
│  - SAS連携（PROC JSON / PROC EXPORT）          │
│  - 共有フォルダ I/O                             │
│  - 認証連携（AD等）                             │
└─────────────────────────────────────────────────┘
                       ↑
                  [SAS バッチ実行]
```

**重要な設計原則**:

- **Domain 層は SAS を知らない** — 「指標」「帳票」「集計単位」という業務概念だけを扱う
- SASをやめて別のデータソースに切り替えてもDomain層は変更不要
- → 将来BigQuery、Snowflake、PostgreSQLに置き換えても他層への影響を最小化できる

### 2.2 ユビキタス言語

開発者と業務担当者が共通で使う語彙を定義し、コード・UIに一貫して反映する。

| 用語 | 意味 |
|------|------|
| 帳票 (Report) | 既存のExcel帳票に対応する単位。1つのテーマ・1つの目的を持つ |
| ダッシュボード (Dashboard) | 複数の指標を1画面に配したWeb版の帳票 |
| 指標 (Metric / KPI) | 1つの数値・グラフ・表として可視化される値 |
| 断面 (Snapshot) | ある時点でのデータ状態（日次断面、月次断面など） |
| 集計単位 (Aggregation Unit) | 顧客 / 部門 / 商品 / 時系列など、集計の切り口 |
| データセット (Dataset) | SASから出力された、ダッシュボードが消費するデータの単位 |

### 2.3 SASとWeb層の「契約」

SASがWebに渡すJSONには、必ず `meta` ブロックを含める。
これがSAS側とWeb側の境界（Anti-Corruption Layer）となる契約である。

```json
{
  "meta": {
    "report_name": "月次売上",
    "generated_at": "2026-05-19T07:00:00",
    "row_count": 48
  },
  "data": [
    { "category": "食品", "yyyymm": "202601", "amount": 1234567 }
  ]
}
```

## 3. SAS → ブラウザの連携パターン

3つの選択肢を提示する。本リポジトリは案A・案Bを実装している。

### 案A：静的ファイル配信パターン（最小構成）

```
SAS Job → PROC JSON でJSON出力 → 共有フォルダ → IIS等の静的配信
                                                  ↓
                                          ブラウザ（HTML/JS）が直接読む
```

**メリット**: バックエンドコード不要、承認が下りやすい、新規ランタイムなし、障害ポイントが少ない
**デメリット**: リアルタイム性なし、複雑なフィルタはフロント任せ

### 案B：軽量Web API パターン

```
SAS Job → JSON / CSV → 共有フォルダ
                          ↓
                    軽量サーバ（PowerShell / Python / Node）
                          ↓
                    REST API → ブラウザ
```

**メリット**: 認証連携、動的フィルタ・集計、データソースを差し替えやすい
**デメリット**: バックエンド言語の許可が必要、サーバ運用増

### 案C：DB中継パターン

```
SAS Job → PROC EXPORT → DB（SQL Server等）
                          ↓
                    軽量Web API → ブラウザ
```

**メリット**: 再利用性、インデックスでパフォーマンス、履歴管理
**デメリット**: DB導入・運用負荷、承認ハードルが最も高い

### 選定指針

| 案 | 想定規模 | 開発許可ハードル | 推奨フェーズ |
|----|----------|----------------|------------|
| A | 帳票数 〜20、データ量小 | 低 | Phase 1（即着手） |
| B | 帳票数 〜100、動的操作多い | 中 | Phase 2 |
| C | 全社展開、複数システム連携 | 高 | Phase 3 |

**まずは案A**で一本通す。需要が確認できたら案Bに昇格する戦略が現実的。

## 4. レベル別の構成

実装の難易度・運用負荷で3レベルに分けて段階的に昇格する。

### レベル0：超ミニマム（file:// で開く）

| 観点 | 内容 |
|------|------|
| メリット | 即日着手 / 承認不要 / コストゼロ / 障害要素ゼロ / フォルダ持ち運び可 |
| デメリット | 複数人同時参照に弱い / 自動更新は別途 / URL共有不可 |
| 向く場面 | 自分用・少人数、PoC、業務担当者との合意形成段階 |

→ [docs/quickstart-level0.md](quickstart-level0.md)

### レベル1：PowerShell HTTPサーバ

| 観点 | 内容 |
|------|------|
| メリット | URLで共有可 / 複数人同時アクセス / 自動更新可能 / 追加ソフト不要 / 承認ハードル低 / 後から拡張しやすい |
| デメリット | サーバー常時稼働必要 / 大量同時接続には弱い / 認証は簡易のみ / 監視は自作 |
| 向く場面 | 5〜20人の部門利用、日次/週次の自動更新が必要、本格運用前のステップ |

→ [docs/level1-guide.md](level1-guide.md)

### レベル2：IIS + AD認証

| 観点 | 内容 |
|------|------|
| メリット | 本格運用耐性 / AD認証でシームレス / IT既存運用に乗る / HTTPS容易 / 監査ログ |
| デメリット | 承認・設定の工数大 / 変更管理が重くなる / 障害責任が重い |
| 向く場面 | 全社・複数部門展開、機密データ、監査要件あり、長期運用確定 |

レベル2は本リポジトリの対象外だが、レベル1で構築したHTML/JS/SASスクリプト/データスキーマは**そのまま再利用できる**設計になっている。

## 5. SAS側のテクニック要点

### 主要なSAS出力方法

| 方法 | 出力形式 | 特徴 |
|------|---------|------|
| `PROC JSON` | JSON | Web連携の本命。9.4以降標準 |
| `PROC EXPORT` | CSV / Excel | 既存帳票互換。シンプル |
| `ODS HTML` | HTML | 静的HTML出力 |
| `ODS EXCEL` | Excel | 高度なExcel出力 |

### 推奨パターン

- **集計はSASで完結させる** — SASの強みを活かす
- **整形・可視化はフロントエンドに寄せる** — SASは「データ提供者」に徹する
- → 境界: SASは「集計済みデータセット + メタデータ」をJSONで出すところまで

## 6. PowerShellがカバーできる範囲

PowerShellがあれば、以下はすべてPowerShellで完結できる。

| 役割 | 実現方法 |
|------|----------|
| SAS実行 | `Start-Process sas.exe -ArgumentList "-SYSIN script.sas"` |
| スケジュール実行 | Windowsタスクスケジューラから `.ps1` を呼ぶ |
| データ変換 | `ConvertFrom-Json` / `ConvertTo-Json` / `Import-Csv` / `Export-Csv` |
| ファイル配置 | コピー、リネーム、世代管理 |
| HTTPサーバ | `System.Net.HttpListener` |
| AD認証 | `[System.DirectoryServices.AccountManagement]` |
| 通知 | `Send-MailMessage` / Teams Webhookへのhttp POST |
| エラーハンドリング | try-catch / `Write-EventLog` |

→ PowerShellが「グルー言語」兼「簡易バックエンド」になる。これが本構成の現実的な土台。

## 7. リスクと対策

| リスク | 影響 | 対策 |
|--------|------|------|
| 制約発見が後手に回り、設計やり直し | 大 | Phase 0で確定する |
| SAS担当者・業務担当者を巻き込めない | 大 | 早期にPoC見せて巻き込む |
| 既存Excel帳票への愛着で移行が進まない | 中 | 並行運用期間を設ける |
| ブラウザの社内標準がIE系で表現制限 | 中 | Phase 0でブラウザを確認 |
| データ量が想定以上で重くなる | 中 | データの間引き、集計済み出力 |
| SASバッチ失敗時のリカバリ | 小 | バッチログ出力＋失敗通知 |

## 8. 「無理に本格を作らない」原則

内製ツールでは、**過剰開発が最大の失敗要因** になる。

- まずレベル0で1帳票PoC → 業務担当者の反応を見る
- 反応が良ければレベル1へ昇格（自動更新・URL共有を追加）
- 部門を超えて要望が出たらレベル2へ昇格

DDDで関心を分離していれば、各レベル間は**作り直しではなく積み増し**で進められる。最初から本格構成を作ると「使われないシステムを作って終わる」リスクが大きい。
